Unit 1

Introduction to Cloud Computing

Cloud computing delivers on‑demand computing resources (compute, storage, networking, platforms, software) over the Internet with pay‑as‑you‑go pricing.

Key Characteristics

  • On‑demand self‑service
  • Broad network access (any device)
  • Resource pooling (multi‑tenancy)
  • Rapid elasticity (scale up/down)
  • Measured service (metering & billing)

Benefits

  • Lower upfront cost, operational expenditure
  • Faster time‑to‑market
  • High availability & global reach
  • Focus on core app, not hardware

Unit 2

Service & Deployment Models

Service Models

  • IaaS — VMs, storage, networks. You manage OS upward.
  • PaaS — Runtime + managed platform. You deploy code.
  • SaaS — Complete apps delivered via browser/API.
  • FaaS — Event‑driven functions (serverless).

Ctrl ↑ ─────────────────────── You App   ⟶ SaaS         │ Code  ⟶ PaaS         │ OS    ⟶ IaaS         │ Events⟶ FaaS         │ Infra ↓ ───────────────────── Cloud

Deployment Models

  • Public (AWS/Azure/GCP) — shared, scalable
  • Private — dedicated, compliance‑heavy
  • Hybrid — on‑prem + public cloud
  • Multi‑cloud — mix of providers
Tip: Exams often ask who manages what in each model.
Unit 3

Virtualization & Containers

Virtualization

  • Hypervisors: Type‑1 (bare‑metal), Type‑2 (hosted)
  • VMs provide strong isolation; slower boot
  • Live migration, snapshots, templates

Containers

  • Share host OS kernel; fast, lightweight
  • Tools: Docker, Podman
  • Orchestration: Kubernetes (pods, services, deployments)
Example — Dockerfile 

Minimal Node.js container

FROM node:20-alpine WORKDIR /app COPY package*.json ./ RUN npm ci --only=production COPY . . EXPOSE 3000 CMD ["node", "server.js"]
Unit 4

Cloud Architecture & Scaling

  • Stateless app tiers behind Load Balancers
  • Horizontal scaling via auto‑scaling groups
  • Decouple with queues & pub/sub (e.g., SQS, Pub/Sub)
  • Cache frequently accessed data (CDN, Redis)
  • Design for failure; multi‑AZ, backups, DR
Diagram — 3‑Tier Cloud 

Users → CDN → LB → App Tier → Queue → Workers ↓ DB Read Replicas
Unit 5

Storage & Databases

Storage Types

  • Object: S3/Blob — scalable, eventual consistency
  • Block: EBS/Persistent Disk — low‑latency volumes
  • File: EFS/Filestore — shared POSIX file systems
  • Cold/Archive: Glacier — low cost, high retrieval time

Databases

  • Relational (RDS/Cloud SQL): ACID, SQL
  • NoSQL (Dynamo/Bigtable/Firestore): scale, flexible
  • Data Warehouses (Redshift/BigQuery/Synapse)
  • Backup, PITR, read replicas, sharding
Tip: Choose storage based on access pattern, latency, consistency, and cost.

Unit 7

Security, Compliance & IAM

  • Shared responsibility model (provider vs customer)
  • Identity & Access Management (principle of least privilege)
  • Encryption at rest & in transit; KMS/Key Vault/Cloud KMS
  • Secrets management (Vault/Secrets Manager)
  • Compliance: ISO 27001, SOC 2, GDPR, PCI‑DSS
  • Incident response: detect → contain → eradicate → recover
Exam gotcha: Don’t put secrets in AMIs/containers; use instance roles/service accounts.

Monitoring

Cheat‑Sheet

Cheat‑Sheet & Definitions

  • Multi‑AZ: Deploy across two or more availability zones.
  • Idempotent: Same action can run repeatedly without side effects.
  • Ephemeral: Temporary, non‑persistent storage/instances.
  • Blue/Green: Two environments; switch traffic after verification.
  • Autoscaling: Adjust instance count based on metrics.
  • RPO/RTO: Data loss window / recovery time objective.
  • CAP: Consistency, Availability, Partition tolerance (choose two).
